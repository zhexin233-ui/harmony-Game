# hwgame 页面改版设计文档

- **日期**：2026-04-20
- **范围**：5 个页面的 UI 改版设计
- **目标平台**：HarmonyOS NEXT / uni-app x（UVue + UTS）
- **设计基线**：仓库内 `ui 设计/` 目录中的 5 份页面设计稿

---

## 1. 目标概述

本次任务是在不改变现有核心玩法、路由主干和双主题能力的前提下，对应用的 5 个页面进行一次成体系的界面改版。改版后的页面需要明显贴近 `ui 设计/` 提供的视觉方向，重点落地到 `Q 版卡通` 主题；`霓虹电玩` 主题不重画，但必须保持可用、可读和不破版。

本次设计聚焦“高还原但可维护”的落地路径：

- 只改 5 个目标页面，不扩散到具体游戏页、结算页、玩家名字页。
- 保留现有真实业务流、真实 store 和真实路由边界。
- 视觉上尽量吸收设计稿中的品牌感、卡片层次、顶部栏、底部导航、装饰背景和图片卡片语言。
- 工程上优先复用现有主题机制、组件体系和页面状态流，不为了还原度把结构写散。

## 2. 已确认约束

### 2.1 范围约束

- 本次一次性覆盖 5 个页面：主页、人数选择、游戏列表、主题选择、惩罚列表。
- 不拆成“先主链路、后补侧链”的分阶段视觉改版。

### 2.2 主题约束

- 保留双主题能力。
- 仅将原有 `Q 版卡通` 主题升级为新设计稿风格。
- `霓虹电玩` 主题继续保留现有定位，只做兼容性适配，不做同等级重绘。

### 2.3 设计落地约束

- 采用“高还原”路线，尽量保留设计稿中的视觉组织方式。
- 设计稿中不存在于当前产品能力内的入口，优先映射到现有真实页面；无法映射时保留视觉并给出占位反馈。
- 游戏列表优先使用仓库内现有本地图片素材，不依赖运行时远程资源。
- 工程优先级为“组件与主题体系尽量复用”，允许适度页面级定制，但不接受整体结构失控。

### 2.4 文案与语言约束

- 页面文案统一使用中文。
- 设计稿中的英文标题、标签、按钮文案全部改写为中文表达。
- 不引入新的英文导航体系或英文占位页面。

## 3. 目标页面与现有页面映射

本次改版不新增业务页面，直接映射到现有路由：

- `ui 设计/主页` → `pages/home/index.uvue`
- `ui 设计/人数选择` → `pages/lobby/player-count.uvue`
- `ui 设计/游戏列表` → `pages/lobby/games.uvue`
- `ui 设计/主题选择` → `pages/onboarding/index.uvue` 与 `pages/settings/index.uvue`
- `ui 设计/惩罚列表` → `pages/punishment/index.uvue`

说明：

- `onboarding` 与 `settings` 共用同一套主题卡片视觉语言，但语义不同。
- 主流程页面为：首页 → 人数选择 → 游戏列表。
- 侧链页面为：设置、惩罚列表。
- 本次不重做 `pages/lobby/player-names.uvue`、各游戏页和结果页，但要保证改版页面跳转到这些页面时流程仍自然。

## 4. 信息架构与导航规则

### 4.1 真实流程保持不变

- 首次启动：`onboarding -> home`
- 首页主入口：`home -> player-count`
- 人数选择：进入 `player-names` 或直接 `games`
- 游戏列表：进入具体游戏页
- 首页次入口：进入 `punishment`、`settings`
- 设置页内保留主题切换
- 惩罚页保留新增、编辑、删除、启用/停用能力

### 4.2 顶部栏策略

这 5 个页面统一采用页面内自绘顶部栏，不依赖系统导航栏展示标题。顶部栏承担以下职责：

- 返回按钮
- 当前页面标题或品牌标题
- 右侧辅助入口，例如头像位、快捷设置位
- 半透明浅色浮层或轻量磨砂视觉

统一采用自绘顶部栏的原因：

- 设计稿中顶部结构个性强，系统导航栏难以承载。
- 有利于 5 个页面在品牌感上形成一致体验。
- 可以将顶部图标、标题、品牌字样和右侧入口统一进同一视觉骨架。

### 4.3 底部导航策略

底部导航将作为统一的页面视觉壳存在于首页、游戏列表、设置页、惩罚页中。其行为规则如下：

- 表达顶层页面切换，而不是多级返回历史。
- 当前页为激活态，点击后不重复入栈。
- 可映射项直接跳转现有真实页面。
- 不可映射项保留视觉结构，点击后仅弹中文占位提示。

建议映射：

- `Home` → 首页
- `Games` → 游戏列表或主开始链路
- `Penalty / Chaos` → 惩罚列表
- `Settings / Profile` → 设置页

本次不为 `Social`、`Store` 等虚构能力新建承载页。

### 4.4 返回规则

- 顶部栏返回优先执行 `navigateBack`。
- 若当前页已处于顶层栈或返回失败，则回到首页。
- 底部导航切页不叠加为深层历史栈。

## 5. 主题系统设计

### 5.1 总体策略

- 保留现有 `cartoon / neon` 双主题结构。
- 将 `cartoon` 升级为新糖果粉设计风。
- `neon` 不重绘，只做兼容性映射，保证新页面结构在深色语境下可读、可用。

### 5.2 Token 扩展原则

现有主题 token 偏基础功能色，不足以承载新页面的卡片层次与导航容器。需要在保留现有基础 token 的同时，增加页面级视觉 token，供共享组件与页面壳层使用。

保留的基础 token 类型：

- 页面背景
- 主色、辅色、强调色
- 主文本、次文本
- 成功、危险、警示类颜色

新增的页面级 token 类型：

- 顶部栏背景
- 底部导航背景
- 激活态胶囊背景
- 卡片背景层级 1 / 2 / 3
- 图片卡片底色
- 浅边框色
- 轻阴影色
- 装饰性圆斑或光斑色
- 占位入口容器底色

### 5.3 兼容性要求

新增 token 在两个主题下都必须有定义。

兼容要求：

- `cartoon` 下完整表达糖果粉新视觉。
- `neon` 下文字对比度充足，不出现浅底浅字或深底深字问题。
- 共享组件内部不直接写死粉色值，统一通过 token 读取样式。

## 6. 共享视觉骨架与组件策略

### 6.1 设计原则

不构建一个吞掉所有页面的大型壳组件，而是抽取轻量共享视觉原语，供页面自由组合。原因如下：

- 5 个页面布局差异较大。
- 高还原设计稿需要保留页面个性。
- 工程上更适合“共享视觉部件 + 页面级结构”的方式。

### 6.2 共享视觉原语

建议抽取以下共享部件或样式规范：

- 顶部栏组件：返回、标题、右侧图标位
- 底部导航组件：圆角悬浮底栏、激活态胶囊按钮、导航映射规则
- 主题卡组件：用于 onboarding 与 settings
- 图片游戏卡组件：用于游戏列表
- 胶囊标签组件：用于人数、分类、状态、来源标签
- 页面装饰背景样式：柔和渐变底、光斑、浮动装饰元素

### 6.3 现有组件的处理策略

- `AppButton.uvue` 保留，升级为可承载新版主按钮与次按钮视觉。
- `GameTile.uvue` 不继续直接承担新版游戏卡片，原因是新版卡片需要图片、角标、提示和禁用态层次，现有抽象过轻。
- 页面允许保留局部结构，不强行把所有设计稿细节塞进通用组件。

## 7. 页面级设计

### 7.1 首页 `pages/home/index.uvue`

目标：落地 `ui 设计/主页` 的品牌入口感，作为应用的统一首页。

结构：

- 顶部自绘 Top Bar
- 中央大标题与副标题区
- 三个主入口卡片
- 页面背景装饰元素
- 底部悬浮导航栏

入口映射：

- 开始玩 → `pages/lobby/player-count`
- 惩罚列表 → `pages/punishment/index`
- 设置 → `pages/settings/index`
- 顶部头像入口 → 设置页

视觉要求：

- 明显的粉色糖果风背景与大标题
- 入口卡片之间有层次差异
- 保留设计稿中的漂浮图标、柔和阴影和圆角体系

边界：

- 不引入真实账号体系
- 不新增个人中心页

### 7.2 人数选择页 `pages/lobby/player-count.uvue`

目标：落地 `ui 设计/人数选择` 的中心数字选择器结构。

结构：

- 顶部返回栏
- 标题与副标题
- 中央大数字显示
- 左右加减按钮
- 一条主按钮 + 一条次级文字入口

交互：

- 人数范围固定 2 到 8
- 默认值使用“会话中已有人数优先，否则为 4”
- 主按钮：进入名字输入页
- 次级文字入口：跳过名字，直接进入游戏列表

视觉要求：

- 大数字居中
- 加减按钮为圆形高强调按钮
- 保留设计稿中柔和背景光斑和中心聚焦感

### 7.3 游戏列表页 `pages/lobby/games.uvue`

目标：高还原 `ui 设计/游戏列表` 的“分类 + 图片卡片”布局。

结构：

- 顶部自绘栏
- 顶部分类过滤条：全部 / 运气派 / 实力派
- 两大分组列表
- 图片游戏卡片网格
- 底部悬浮导航栏

卡片内容：

- 游戏图片
- 中文名称
- 简短副提示
- 人数角标
- 不可玩提示

数据与交互：

- 游戏数据仍来源于现有元数据定义
- 分类过滤为当前页局部 UI 状态，不写入 store
- 不可玩的游戏保留展示但禁用点击
- 人数不匹配时显示明确限制文案

图片策略：

- 优先使用 `ui 设计/img` 中的本地素材
- 如果素材缺失、加载失败或不适配，则回退到 emoji / 图标占位
- 不允许因单张图片异常导致页面不可用

### 7.4 主题选择 `pages/onboarding/index.uvue` 与 `pages/settings/index.uvue`

目标：把 `ui 设计/主题选择` 的大卡片语言统一落到首次引导与设置页。

`onboarding` 结构：

- 顶部轻量品牌栏或无系统导航的沉浸式标题区
- 两张主题大卡片
- 用户点击卡片后立即应用主题并进入首页

`settings` 结构：

- 顶部自绘栏
- 主题切换区作为最主要视觉模块
- 下方保留音效、振动、关于信息等已有设置项
- 页面底部统一导航

主题呈现规则：

- `Q 版卡通` 卡片展示新版糖果风
- `霓虹电玩` 卡片展示原有深色赛博感
- 当前选中态必须明确，但不依赖复杂动画

### 7.5 惩罚列表页 `pages/punishment/index.uvue`

目标：落地 `ui 设计/惩罚列表` 的“顶部新增入口 + 卡片化列表”结构，同时保留现有惩罚库能力。

结构：

- 顶部自绘栏
- `新增惩罚` hero 卡片
- 惩罚规则列表
- 每条规则的独立卡片
- 底部统一导航

每条卡片内容：

- 规则正文
- 内置 / 自定义标签
- 启用开关
- 编辑入口
- 删除入口
- 左侧状态色条

状态表现：

- 禁用条目使用弱化对比 + 删除线或灰化处理
- 空状态下仍保留顶部 hero 卡片与统一页面骨架，不切换到完全不同的模板

不扩展的能力：

- 不新增分类系统
- 不新增批量管理
- 不改变现有惩罚数据结构

## 8. 交互规则与占位策略

### 8.1 点击反馈

- 主按钮、卡片、导航项统一使用轻量点击反馈，例如轻微缩放、透明度变化。
- 不使用长时、重依赖的复杂动画。

### 8.2 占位入口规则

- 能映射到现有页面的入口必须直接跳转。
- 无法映射的入口保留视觉结构，但点击后仅弹中文提示，例如“功能建设中，稍后开放”。
- 不创建新的空白页面来承载占位入口。

### 8.3 主题切换规则

- `onboarding` 中点击主题卡后立即写入设置、标记已引导并进入首页。
- `settings` 中切换主题后当前页面立即响应主题变化。
- 不增加二次确认弹窗。

### 8.4 游戏列表禁用态规则

- 因人数不匹配而不可玩的游戏卡片仍显示在原位。
- 禁用态下提供清晰的人数限制文案。
- 禁用态卡片不响应进入游戏的点击。

### 8.5 惩罚列表操作优先级

- 启用 / 停用：轻操作，直接生效
- 编辑：进入现有编辑页
- 删除：保留确认弹窗

---

## 9. 异常处理与降级规则

### 9.1 图片与资源降级

- 游戏封面图加载失败时，自动降级到 emoji / 图标展示。
- 不允许因图片异常导致列表错位、空白或无法点击。

### 9.2 设置与主题降级

- 设置读取失败时回退到默认设置。
- 缺失主题变量时仍应保证页面使用安全的默认颜色，不出现不可读文本。

### 9.3 惩罚数据降级

- 惩罚数据异常时回退到内置规则或空状态视图。
- 页面结构保持可操作，不因数据异常崩溃。

### 9.4 路由与返回降级

- 返回失败时回到首页。
- 顶层导航切换不依赖复杂历史栈。

## 10. 测试策略

### 10.1 自动化测试重点

本次不引入重量级页面快照测试，重点补足受改版影响的纯逻辑和主题测试：

- 扩展主题 token 完整性测试
- 扩展主题变量写入测试
- 补充导航映射与页面纯逻辑辅助函数测试
- 补充人数按钮边界逻辑测试
- 补充游戏图片路径选择与降级策略测试

### 10.2 现有测试不允许回归

以下既有测试必须继续通过：

- 游戏逻辑测试
- `settings / session / punishment` store 测试
- 主题 token 与应用测试
- 原生能力相关测试

### 10.3 测试时长约束

- 单元测试总耗时需控制在 60 秒以内。
- 本次页面改版不得引入明显拖慢测试套件的额外依赖。

## 11. 验收标准

### 11.1 视觉验收

- `cartoon` 主题下，5 个目标页面整体视觉明显贴近设计稿。
- 首页、人数页、游戏列表、主题选择、惩罚页在视觉风格上保持统一。
- 中文文案替换后版式仍稳定，不出现明显溢出与错位。

### 11.2 功能验收

- 5 个页面之间的主要跳转完整可用。
- 底部导航可映射项都能跳转到真实页面。
- 不可映射项只做占位提示，不出现空白页面。
- 人数范围被严格限制在 2 至 8。
- 游戏列表不可玩项保留可见并明确禁用。
- 惩罚页空状态和非空状态都保持统一视觉骨架。
- 主题切换立即生效。

### 11.3 兼容验收

- `neon` 主题下不破版、不出现不可读文本。
- 资源缺失时页面可降级显示。
- 不因自绘顶部栏与底部导航破坏原有业务流。

## 12. 实施边界

### 12.1 允许的改动

- 扩展主题 token
- 新增少量共享 UI 组件
- 将 5 个页面切换为自绘顶部栏
- 为顶层页面加入统一底部导航
- 对页面结构进行重排
- 将页面映射逻辑抽为纯函数便于测试

### 12.2 不在本次范围内的改动

- 不新增真实业务页面承接 `Social / Store` 等入口
- 不改造 `pages/lobby/player-names.uvue`
- 不改造具体游戏页与结果页
- 不改动游戏核心规则
- 不替换 Pinia 或主题体系的总体架构
- 不引入远程运行时资源依赖

---

## 13. 推荐实施路径

推荐采用“主题 Token 升级 + 页面局部重构”的实现策略：

1. 先扩展 `cartoon / neon` 主题 token，保证新视觉骨架有稳定的样式输入。
2. 再补充顶部栏、底部导航、主题卡、游戏卡等轻量共享组件。
3. 随后逐页改造 5 个目标页面，优先完成主链路，再补设置与惩罚页。
4. 在实现过程中用小函数承载映射与降级逻辑，补对应单测。

该路径兼顾三点：

- 满足高还原设计目标
- 尽量复用现有工程结构
- 控制后续维护成本

- 现有业务流与 store 行为未被破坏
